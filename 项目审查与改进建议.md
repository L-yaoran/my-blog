# 项目审查与改进建议

> 基于对整个 AI幻世录·GeoLib 项目的全面审查，整理如下问题分析与改进方向。

---

## 一、架构层面

### 1.1 双首页系统

**问题：** `hub.html`（当前首页）和 `index.html`（博客列表页）两套页面并存，各自有独立的导航栏和页脚。`hub.html` 不继承 `base.html`，内联了 277 行 CSS 和完整的导航/页脚 HTML。`index.html` 虽然未被使用（已移入回收站），但反映了架构上的不一致性。

**影响：** 修改导航栏或页脚需要同时维护两套 HTML。

**改进建议：**

| 方案 | 复杂度 | 效果 |
|------|--------|------|
| 让 hub.html 继承 base.html | 中 | 统一导航/页脚，删除内联 CSS |
| 将 hub 内联 CSS 抽到 style.css | 低 | 至少让 CSS 不重复 |
| 保持现状（hub 作为独立"单页应用"） | 无 | 灵活但维护成本高 |

### 1.2 练习页面的内联结构

**问题：** 虽然已抽离了公共 CSS（`quiz.css`）和 JS（`quiz-engine.js`），但每个练习页面的 HTML 框架（导航栏、工具栏、历史面板、答题详情弹窗）仍然是重复的。6 个文件里各有一套完全相同的 ~120 行 HTML 骨架。

**影响：** 修改工具栏布局或新增功能按钮需要改 6 个文件。

**改进建议：**
- 用 Jinja2 模板或服务端 include 生成练习页面的公共骨架
- 或者在生成脚本中预定义骨架模板，统一渲染

### 1.3 没有版本控制

**问题：** 项目根目录有 `.git` 目录但没有 `.gitignore` 跟踪 `/回收站` 目录，部分文件被误删后无法恢复。

**建议：**
- 将整个项目纳入 git 跟踪
- 在 `.gitignore` 中排除 `回收站/`、`__pycache__/`、`venv/`
- 每次重大改动前提交一个版本

## 二、前端层面

### 2.1 CSS 变量冲突风险

**问题：** `style.css`（博客）和 `quiz.css`（练习）各自定义了 `:root` CSS 变量（`--blue`、`--gold` 等），值相同但分散在两处。如果未来要统一调整主题色，需改两个文件。

**建议：** 将公共 CSS 变量提取到 `variables.css`，或让 `quiz.css` 导入 `style.css` 的变量。

### 2.2 练习页面 localStorage 数据没有备份机制

**问题：** 用户的练习进度和做题记录全部存储在浏览器 `localStorage` 中，一旦清除浏览器缓存、更换设备或重装系统，所有数据丢失。

**建议：**
- 添加**导出/导入**功能：**暂存**和**加载**按钮旁边增加「导出数据」和「导入数据」，文件为 JSON
- 这样用户可以手动备份练习进度

### 2.3 移动端体验有待优化

**问题：** 练习页面的工具栏在移动端屏幕较窄时，两行布局中的按钮会挤压或换行，历史面板的绝对定位可能超出屏幕。

**建议：**
- 在 `quiz.css` 中增加移动端断点（如 480px），按钮改为图标+文字的简洁模式
- 历史面板在移动端改为全屏弹窗而非绝对定位下拉

### 2.4 代码高亮不支持深色模式

**问题：** `quiz.css` 中的代码高亮只适配了浅色主题，如果用户浏览器启用深色模式，代码块背景和文字颜色可能不协调。

**建议：** 添加 `@media (prefers-color-scheme: dark)` 媒体查询，提供深色主题的代码高亮配色。

## 三、后端层面

### 3.1 练习页面不是通过 Flask 路由渲染的

**问题：** 练习页面是放在 `static/` 目录下的纯静态 HTML，不是通过 Flask 的 `render_template()` 渲染的。这意味着：
- 不能使用 Jinja2 模板继承
- 不能利用 Flask 的静态文件版本控制
- URL 路径是 `/static/python_basic_test_001.html` 而非更友好的 `/practice/001`

**建议：**
- **低优先级：** 将练习页面迁移到 `templates/`，通过 Flask 路由渲染，用 Jinja2 include 公共骨架
- 这样 URL 可以改为 `/practice/001/` 等更友好的格式

### 3.2 没有单元测试

**问题：** 整个项目没有测试代码，依赖手动测试。之前多次出现工具栏布局错乱、JS 函数重复等问题。

**建议：**
- 为 `app.py` 的路由添加 Flask 测试客户端的基本请求测试
- 为 `quiz-engine.js` 的关键函数（`normalize`、`collectAnswers` 等）添加简单的冒烟测试

### 3.3 Nginx 静态缓存策略简单

**问题：** 当前 Nginx 配置对所有静态文件设置 `expires 1h`。CSS/JS 更新后用户可能看到缓存版本，虽然使用了 `?v=timestamp` 参数，但只在 `base.html` 中用了，练习页面没有缓存破坏机制。

**建议：**
- 在 `quiz.css` 和 `quiz-engine.js` 引用 URL 后添加版本参数，如 `/static/css/quiz.css?v=20260601`
- 或考虑使用文件内容 hash 作为版本号

## 四、内容管理层面

### 4.1 博客文章数量较少

**问题：** 目前只有 3 篇博客文章。作为技术博客，内容密度较低。

**建议：** 可以规划的内容方向：
- 练习页面配套的**知识点讲解文章**（如「Python 网络编程入门」「多进程与多线程对比」）
- **项目实战系列**（如「从零搭建 Flask 博客」「Nginx 配置踩坑记录」）
- **学习笔记整理**（如「闭包与装饰器深度理解」）

### 4.2 练习页面内容固化

**问题：** 练习页面的题目数据直接嵌入 HTML，修改题目需要编辑 HTML 文件，不适合非技术人员。

**建议：** 如果未来练习数量大幅增长，可考虑将题目数据以 JSON 格式存储，通过 JS 动态加载渲染。

## 五、可维护性层面

### 5.1 生成脚本在回收站

**问题：** `generate_day04.py`、`generate_day05_1.py`、`generate_day05_2.py` 等生成脚本都放在了回收站，但它们是理解练习页面生成逻辑的重要资产。

**建议：** 将生成脚本移到一个 `scripts/` 目录并保留，添加注释说明用途和使用方式。

### 5.2 CLAUDE.md 已过时

**问题：** 项目根目录的 `CLAUDE.md` 记录了项目信息，但内容未及时更新（如缺少 005-2 练习、缺少 `quiz.css`/`quiz-engine.js` 等）。

**建议：** 同步更新 `CLAUDE.md`，保持与项目实际结构一致。

### 5.3 练习页面编号混用 `-` 和数字

**问题：** 当前编号系统是 `001, 002, 003, 004, 005-1, 005-2`。`005-1` 和 `005-2` 的命名方式与前面的纯数字不一致，且 JS 中处理练习编号的逻辑（如导航高亮）需要特殊处理带 `-` 的编号。

**建议：** 如果后续还有 Day06+的练习，建议统一编号规则。例如全部采用 `NNN` 格式（`006`, `007` 等），或全部采用 `NNN-M` 格式（`006-1`, `006-2`），避免混合。

## 七、总结

这个项目虽然小，但五脏俱全——有内容管理、有交互式练习、有完整的部署链路。最大的优势是 **"轻"**：无数据库、纯静态内容、部署简单。 

核心改进方向可以概括为 **"统一与备份"**：

1. **统一** — 模板统一（hub/base）、样式统一（CSS 变量）、编号规则统一
2. **备份** — 练习进度可导出、版本控制纳入 git、生成脚本保留

目前 6 个练习页面、3 篇博客文章的状态对于个人学习站点来说已经是一个不错的起点。优先做 P0 和 P1 的改进即可显著提升维护体验。
